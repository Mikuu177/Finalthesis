# 项目描述（毕业设计仓库：Finalthesis）

> 本文档基于仓库**代码与配置文件**的阅读结果整理（不依赖 README）。

## 1. 仓库结构总览

- **`frontend/`**：静态前端页面（HTML/CSS/JS），提供“多模型回答对比”的交互界面。
- **`backend/`**：FastAPI 后端服务，提供统一接口并发调用多个大模型（OpenAI SDK 的 Async 客户端）。
- **`temporal-reasoning-benchmark/`**：时间推理基准测试工具链（数据→prompt→调用→抽取→归一化→评分→输出汇总）。
- **`outlineAndPlan_201691597_WenhaoNie.docx`**：毕业设计相关的提纲/计划文档（本 `项目描述.md` 当前未引用其中内容，避免引入未核实信息）。

## 2. Web：LLM 对比平台（frontend + backend）

### 2.1 前端（`frontend/`）

- **入口**：`frontend/index.html`
- **交互逻辑**：`frontend/script.js`
- **样式**：`frontend/style.css`

前端页面包含：
- 一个 `textarea` 供用户输入 prompt
- 一个“提交查询”按钮
- 四个模型结果卡片：`DeepSeek`、`Qwen`、`OpenAI`、`Doubao`

前端请求固定地址：
- `POST http://127.0.0.1:8000/api/query`
- body：`{ "prompt": "..." }`

渲染逻辑：
- 点击提交后先进入 loading 态
- 后端返回 `[{model, response, error?}, ...]` 后，按 `model` 字段映射到对应卡片

### 2.2 后端（`backend/main.py`）

- **框架**：FastAPI
- **接口**：
  - `POST /api/query`：并发查询已配置的模型，返回结果列表
  - `GET /`：健康检查 `{"status": "Backend is running"}`
- **跨域**：CORS 允许所有来源（`allow_origins=["*"]`）

后端并发策略：
- 在 `run_queries` 中使用 `asyncio.gather` 同时调用：
  - `query_deepseek`
  - `query_qwen`
  - `query_openai`
  - `query_doubao`

密钥与客户端初始化：
- 从环境变量读取：
  - `DEEPSEEK_API_KEY`
  - `QWEN_API_KEY`
  - `OPENAI_API_KEY`
  - `DOUBAO_API_KEY`
- 若 key 缺失或包含 `"YOUR_"`，该 client 置为 `None`，调用时返回 `error="API key not configured."`

依赖（`backend/requirements.txt`）：
- `fastapi`
- `uvicorn[standard]`
- `python-dotenv`
- `openai`

## 3. Benchmark：时间推理评测工具链（`temporal-reasoning-benchmark/`）

### 3.1 目标与输出

该子项目提供一个“可复现”的评测运行流程：
- 读取数据集（JSONL，统一 schema）
- 按模板构建 prompt（支持 `sp` / `cot` 与 few-shot）
- 通过 adapter 调用模型（支持 OpenAI 与 OpenAI-compatible provider）
- 从输出中抽取 `FINAL_ANSWER`
- 对答案做归一化与评分
- 写出逐条预测与汇总指标

主要输出（每次 run 生成一个 run_dir）：
- `predictions.jsonl`：逐样本预测记录（包含 prompt、raw、pred、pred_norm、gold、correct 等）
- `summary.csv`：按 model+category 汇总指标（accuracy、contract_rate、extract_rate 等）
- `config_snapshot.yaml`：本次运行的配置快照（config/models/prompts）
- `run_metadata.json`：包含 `run_config_hash`、git commit（尽力获取）、耗时等元信息
- `sample_manifest.jsonl`：本次采样的样本 id/category 清单

### 3.2 运行入口（`src/runner.py`）

- CLI：`python -m src.runner --config <config.yaml> [--models ...] [--prompts ...]`
- 读取：
  - `configs/*.yaml`（本次运行参数）
  - `configs/models.yaml`（模型适配配置）
  - `configs/prompts.yaml`（prompt 模板配置）

核心流程摘要：
- `load_dataset(dataset_path, subset_size=...)`
- 可选过滤与采样：`filter_category` + `sample_size`（随机种子 `seed`）
- 对每个 model、每个 sample：
  - `build_prompt(...)` 生成 `full_prompt`
  - `adapter.generate([...], params, cache_dir, cache_key_extra=sid)` 调用模型（带 cache + retry）
  - `extract_final_answer(raw_text)` 抽取答案（优先最后一行 `FINAL_ANSWER:`）
  - `normalize_model_answer(pred, sample)` 做归一化（含 MCQ 映射、时间/日期片段提取、数值规整）
  - `score(pred_norm, gold, relaxed=False)` 严格匹配
  - 写入 `predictions.jsonl`
- 结束后生成 `summary.csv` 与 metadata 文件

统计的关键指标（summary.csv）：
- **accuracy**：严格匹配正确率
- **contract_rate**：模型输出中是否满足契约（出现 `FINAL_ANSWER:`，并检测最后一行契约）
- **extract_rate / parse_rate**：答案是否被识别为（时间/日期/数值等）可抽取形态
- 形态分布：`time_rate`、`date_rate`、`numeric_rate`、`label_rate`、`text_rate`
- `tz_token_rate`：输出是否包含时区 token（粗粒度 regex）

### 3.3 Prompt 构建（`src/prompt_builder.py`）

- 模板映射来自 `configs/prompts.yaml`：
  - `prompt_mode: sp` 使用 `sp_0shot.txt` / `sp_5shot.txt`
  - `prompt_mode: cot` 使用 `cot_0shot.txt` / `cot_5shot.txt`
  - 对 `cot + 0shot + prompt_version=v2.2` 特判使用 `cot_0shot_v22.txt`

few-shot exemplars：
- 由 `prompts/exemplars.jsonl` 提供
- `_format_exemplar` 会把 exemplar 格式化为：
  - `Question:`
  - `Context:`（如有）
  - `Reasoning:`（仅 cot 模式且 exemplar 有 rationale）
  - `FINAL_ANSWER: <gold>`

类别规则（`category_rules`）会注入 prompt：
- `Hour Adjustment (24h)`：时间加减按 24 小时取模，输出 `HH:MM`
- `Hour Adjustment (12h)`：必要时按 12 小时理解，但输出仍为 24 小时 `HH:MM`
- `Date Computation/Year Shift/Month Shift`：按公历，输出 `YYYY-MM-DD`
- `Time Zone Conversion`：按时区转换方向，输出 `HH:MM`

### 3.4 答案抽取与评分（`src/scorer.py`）

抽取：`extract_final_answer(text)`
- 优先读取“最后一行”是否满足 `FINAL_ANSWER: ...`
- 否则退化为全文最后一次出现的 `FINAL_ANSWER:`
- 再不行取最后一行非空文本

基础评分：`score(pred_raw, gold_raw)`
- `normalize_answer` 后做严格字符串相等

增强评分（v1）：`score_v1(pred_raw, gold_raw, category)`
- `duration`：支持把“1 小时 45 分钟”等解析成分钟对比
- `relation`：对“在前/在后/同时/相邻且不重叠/包含”等做同义归一化与一些宽松匹配

### 3.5 归一化与“输出形状”识别（`src/normalize.py`）

`normalize_model_answer(pred, sample)` 的优先级：
1. **MCQ 映射**：若 `sample.metadata.mcq_options` 存在且 pred 为 A/B/C/D，则映射到选项文本
2. **提取时间片段**：文本中出现 `H:MM`，取最后一次并规范为 `HH:MM`
3. **提取日期片段**：提取最后一次 date-like fragment 并转 `YYYY-MM-DD`
4. **数值归一化**：去空白、千分位逗号，浮点规整

形态识别函数：
- `is_label_answer / is_numeric_answer / is_time_answer / is_date_answer`
- `contains_time_fragment / contains_date_fragment / contains_tz_token`

### 3.6 数据与数据准备

#### 3.6.1 统一数据加载（`src/io_utils.py::load_dataset`）

期望最小 schema：
- `{ id, question, gold, category, metadata(optional) }`

加载后统一为：
- `id`
- `question`（或 `input`）
- `context`（缺省空串）
- `gold`（或 `answer`）
- `category`
- `metadata`（或 `meta`）

#### 3.6.2 示例数据（`data/raw/sample.jsonl`）

- 含 `arithmetic / duration / relation / frequency` 四类中文示例题
- gold 形式包括：日期、时长中文表达、关系描述

#### 3.6.3 TRAM 数据转换脚本（`scripts/prepare_tram.py`）

用途：把 TRAM CSV 转成统一 JSONL，并生成审计文件 `*.audit.json`。

关键点：
- 若答案为 `A/B/C/D` 且能从 CSV 行获得 `Option A-D`：
  - 写入 `metadata.mcq_options`
  - `gold` 会被改写为“正确选项对应的文本答案”（并规整），并记录 `gold_mode=mcq_semantic`
- 否则可能保留 `gold_mode=mcq_label`（仅标签）或 `raw`

### 3.7 模型适配（Adapters）

- `BaseAdapter.generate`：
  - 构造 fingerprint 用于缓存（`cache_dir`）
  - 失败时指数退避重试
  - 返回 `GenerateResult(text, latency, usage, error)`

- `build_adapter`（`src/adapters/__init__.py`）：
  - `provider: openai` → `OpenAIAdapter`
  - `provider: openai_compatible` → `OpenAICompatAdapter(base_url=...)`
  - API key 取 `api_key_env` 指定环境变量，缺省 fallback 到 `OPENAI_API_KEY`

### 3.8 配置文件（Configs）

- `configs/models.yaml`：定义模型 provider/base_url/model/api_key_env
- `configs/prompts.yaml`：定义 prompt 模板文件与 exemplars 路径
- `configs/baseline.yaml`：基线运行示例（sample.jsonl + deepseek_chat）

Pilot 组（用于小规模试验/冒烟）：

### 3.9 运行辅助脚本（Scripts）

- `scripts/run_pilot.py`：
  - 试图定位 repo（当前目录或 Desktop 递归查找）
  - 执行 `python -m src.runner --config <...>`

- `scripts/run_cot0_v21.py`：
  - Windows Desktop 路径写死（`C:\Users\Administrator\Desktop`）
  - 写死 config：`configs\pilot_tram_cot0.yaml`

- `scripts/quick_metrics.py`：
  - 输入：`<run_dir>/predictions.jsonl`
  - 输出：打印 overall 与按 category 的 `parse_rate`、`accuracy`
  - 注意：该脚本读取字段 `row.get('parsed')` 作为 parse 标记，若 `predictions.jsonl` 没有该字段则 parse_rate 可能为 0。

- `scripts/rescore_run.py`：
  - 输入：`<run_dir>/predictions.jsonl`
  - 处理：使用 `src.scorer.score_v1(pred, gold, category)` 进行重评分（增强 duration/relation 的匹配能力）
  - 输出：
    - `summary_v1.csv`
    - `quick_metrics_v1.txt`
    - `failures_v1.jsonl`（所有不正确样本，含 `raw_tail`）
  - 还会更新：`run_metadata.json` 写入 `scoring_version/scoring_notes`
  - 注意：该脚本同样依赖 `row.get('parsed')` 统计 parse_rate。

- `scripts/extract_failures.py`：
  - 输入：`<run_dir>/predictions.jsonl`
  - 输出：在终端展示（各最多 2 条示例）：
    - `Parse Fail`：`parsed` 为 false 的样本
    - `Incorrect Prediction`：`correct` 为 false 的样本
    - `Format Anomaly`：启发式格式异常（例如 gold 是 ISO 日期但 pred 不是；duration 类缺少小时/分钟关键词等）
  - 注意：该脚本判定 parse fail 同样依赖 `row.get('parsed')` 字段。

- `scripts/audit_tz_smoke.py`：
  - 输入：TZ smoke 的 `<run_dir>/predictions.jsonl`
  - 输出：打印一个 JSON 汇总，主要统计：
    - gold 中是否包含 time/date/tz token 的比例
    - gold 是否“纯时间”或“日期+时间”
    - `match_after_time_extract_rate`：对 gold/pred 提取时间片段后是否相等

## 4. 当前已识别的风险点（仅记录，不做修改）

- **字段名不一致风险**：
  - `runner.py` 写入的是 `extract_ok`、`contract_ok`、`correct` 等字段
  - 但 `scripts/quick_metrics.py` 与 `scripts/rescore_run.py` 统计 parse_rate 时读取 `row.get('parsed')`
  - 这会导致 `parse_rate` 统计可能长期为 0（因为 `parsed` 字段缺失）。

- **第三方 API 配置不确定性**：
  - Doubao 的 `base_url`/`model` 在多处标注“请核实”（例如 `configs/models.yaml`，以及后端 `backend/main.py` 也有类似假设）。

- **依赖版本未锁定**：
  - 后端与 benchmark 对 `openai` 版本未做强约束（benchmark 仅 `openai>=1.0.0`），跨机器安装可能造成行为差异。

- **可移植性问题**：
  - `run_cot0_v21.py` 强绑定 Windows Desktop 路径，跨平台/跨目录不可直接复用。

### 5.3 误差分析样例集（每个类别 2–3 条典型失败）

#### 5.3.1 模板（每条一条）

- **Category**: <类别>

- **Gold**: <标准答案>

  - 改进抽取规则（如更宽松的时间片段正则） / 改进归一化（如支持“小时/分钟”中文单位） / 修正时区方向提示词 / 增加算术示例 / 改进关系词同义词表 / 强化格式契约提示 / 其他

#### 5.3.2 示例（虚构，请替换为真实失败样本）

- **Category**: Hour Adjustment (24h)
- **Question**: What is 23:48 - 01:31?
- **Context**: （无）
- **Gold**: 22:17
- **Model Output (raw tail)**:
  ```
  Reasoning: Subtract 1 hour 31 minutes from 23:48.
  23:48 - 1:31 = 22:17
  FINAL_ANSWER: 22:17
  ```
- **归因**: 无（此为正确示例，仅作模板参考）

- **Category**: Duration
- **Question**: 会议从 14:20 到 16:05，持续多长时间？
- **Context**: （无）
- **Gold**: 1 小时 45 分钟
- **Model Output (raw tail)**:
  ```
  16:05 - 14:20 = 1:45
  FINAL_ANSWER: 1:45
  ```
- **归因**: 时间表达归一化错（模型输出“1:45”而非“1 小时 45 分钟”，导致 `is_time_answer` 为 true 但 `normalize_model_answer` 未转为分钟，评分不匹配）
- **修复路径**: 在 `normalize_model_answer` 中增加对“纯时间”到“中文时长”的映射规则，或在评分 v1 中对 duration 类别做“时间差值转分钟”的宽松匹配

- **Category**: Time Zone Conversion
- **Question**: 若 UTC 时间是 08:00，换算成 CST（+8）是几点？
- **Context**: （无）
- **Gold**: 16:00
- **Model Output (raw tail)**:
  ```
  CST = UTC + 8
  08:00 + 8 = 16:00
  FINAL_ANSWER: 00:00
  ```
- **归因**: 算术错（模型误把加法做成取模或误算）
- **修复路径**: 在 prompt 中明确“不取模，直接加 8 小时”，或在 `category_rules` 中加入“时区转换不取模”的提示

> **如何获取**：用 `scripts/extract_failures.py` 抽取失败样本，再人工归因并按上述模板整理。

### 5.4 结构化中间表示（timeline/event）的最小实现计划

> 目标：把自然语言时间描述映射到结构化事件/时间线，便于后续推理与误差归因。

#### 5.4.1 Schema（最小可用）

```yaml
Event:
  id: str          # 事件唯一标识
  description: str # 原始描述（可选）
  time: TimePoint  # 时间点或时间区间
  attributes: dict # 其他属性（如地点、参与者等）

TimePoint:
  type: "point" | "interval"
  value: str       # ISO 8601 时间点（如 "2021-01-04T14:20:00"）或区间（如 "2021-01-04T14:20:00/2021-01-04T16:05:00"）
  timezone: str    # 时区（如 "UTC" / "CST"）
  duration: str    # 若为区间，可附带时长（如 "PT1H45M"）

Timeline:
  events: list[Event]
  relations: list[Relation]  # 事件间关系（before/after/overlap/contains）

Relation:
  from: str        # 事件 id
  to: str          # 事件 id
  type: "before" | "after" | "simultaneous" | "overlap" | "contains"
```

#### 5.4.2 实现步骤（最小可行）

1. **时间片段抽取**：
   - 复用 `normalize.py` 中的 `extract_time_fragment` 与 `extract_date_fragment_to_iso`
   - 对 duration 类，新增 `parse_duration_minutes`（已在 `scorer.py` 实现）并输出 `PT...M` 格式

2. **事件构造**：
   - 对每条样本，用 `question` 与 `context` 抽取所有时间点/区间，生成 `Event` 列表
   - 若为 MCQ，把选项也视为候选事件

3. **关系推断**：
   - 基于 `question` 中的关键词（“在前/在后/同时/相邻/包含”）构造 `Relation`
   - 可用 `scorer.py` 中的 `relation_canonical` 与 `relation_match_v1` 做归一化

4. **推理验证**：
   - 对构造的 `Timeline` 进行一致性检查（如区间重叠、时区转换方向）
   - 若不一致，返回“中间表示冲突”作为误差归因

5. **输出与可视化**（可选）：
   - 把 `Timeline` 序列化为 JSON/YAML，便于脚本分析
   - 可用前端页面渲染简单时间线（可选，非必须）

#### 5.4.3 与现有 pipeline 的集成点

- **前置**：在 `runner.py` 的 `normalize_model_answer` 之后，调用“中间表示构造器”
- **后置**：在 `score_v1` 之前，用结构化表示做“语义宽松匹配”（如 duration 分钟对比）
- **诊断**：在 `extract_failures.py` 中加入“中间表示冲突”的归因类别

> **最小成果**：即使不实现完整推理，仅把“时间片段→结构化表示”做出来，就能显著提升误差归因的粒度（例如区分“抽取失败” vs “推理算术错”）。

## 6. 你可以如何使用本仓库（基于现有代码的“事实描述”）

- **Web 平台用途**：用于交互式输入一个问题，同时观察不同模型的回答差异。
